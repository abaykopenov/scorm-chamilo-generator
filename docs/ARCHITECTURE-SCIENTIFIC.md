# Мультиагентная архитектура автоматизированного проектирования электронных курсов на базе больших языковых моделей (LLM) и RAG-паттернов

**Аннотация**
В данном документе приводится строгое техническое описание распределенной программной архитектуры **SCORM-Chamilo Generator**. Представленная система решает задачу автоматизированной трансформации неструктурированных текстовых массивов (PDF-документов) в интерактивные образовательные модули формата SCORM 1.2 с последующей бесшовной гибридной интеграцией в систему управления обучением (LMS) Chamilo. В основе решения лежит парадигма Retrieval-Augmented Generation (RAG) и концепция локальной мультиагентной оркестрации, обеспечивающей высокую степень защиты данных (Data Privacy) и минимизирующую частоту алгоритмических галлюцинаций.

---

## 1. Введение и постановка проблемы

Традиционная методология разработки электронных курсов (Instructional Design) характеризуется высокими трудозатратами на анализ исходных материалов, структурирование контента и разработку контрольно-измерительных материалов (КИМ). Попытки автоматизации данного процесса посредством "наивного" использования больших языковых моделей (LLM) сопряжены со следующими фундаментальными ограничениями:

1. **Ограничения контекстного окна (Context Boundary Limits):** Невозможность единоразовой обработки объемных документов (сотни страниц) без потери семантической точности.
2. **Алгоритмические галлюцинации (Hallucinations):** Склонность LLM к генерации морфологически корректного, но фактологически неверного контента (False Positives).
3. **Ограничения телеметрии стандарта SCORM:** Инкапсуляция итоговых тестов в SCORM-ассеты приводит к потере детализированной учебной аналитики (Item Response Theory) на стороне LMS.

Проектируемая система решает указанные проблемы посредством внедрения многоуровневого конвейера генерации.

---

## 2. Архитектура подсистемы RAG (Retrieval-Augmented Generation)

Для устранения фактологических искажений в системе реализован конвейер RAG, жестко ограничивающий генеративное пространство LLM извлеченными фактами.

### 2.1. Препроцессинг и очистка (Data Sanitization)
Сырой базис (PDF-документы) конвертируется в текстовый массив и проходит лексическую очистку (`cleanRagText`). Данный модуль применяет эвристические алгоритмы и регулярные выражения для элиминации "мусорных" токенов:
* Исключение метаданных документа (email-адреса, URL, сквозные колонтитулы).
* Восстановление целостности токенов (склейка разорванных переносом слов `word-\npart`), что критически снижает показатель Perplexity при последующем эмбеддинге.

### 2.2. Семантический чанкинг и векторизация
Очищенный текст сегментируется на перекрывающиеся логические фрагменты (chunks). Для каждого сегмента вычисляется плотное векторное представление (Embeddings) большой размерности с использованием специализированной модели-энкодера. Векторы индексируются в высокопроизводительной векторной СУБД **Qdrant**.
Поиск пространственно близких векторов (Cosine Similarity) гарантирует, что LLM получит в контекст запроса исключительно релевантные факты (Information Retrieval).

## 3. Ядро мультиагентной оркестрации (Multi-Agent Engine)

Проблема превышения лимитов вычислительных мощностей локальных GPU решается путем декомпозиции глобальной задачи на узкоспециализированные промпт-цепочки, выполняемые виртуальными агентами под управлением Оркестратора.

### 3.1. Фаза А: Структурный синтез (Planner Agent)
Агент инициирует построение таксономии курса (деревья: модули -> разделы -> SCO -> экраны). Агент извлекает из Qdrant смысловые доминанты исходного текста и генерирует формализованный JSON-граф оглавления курса без заполнения обучающим контентом.

### 3.2. Фаза Б: Генерация контента (Writer Agent)
Процесс наполнения графа происходит путем синхронных итераций. Для каждого терминала графа (экрана) Оркестратор формирует локальный RAG-контекст и делегирует генерацию Писателю. Строгий дизайн системных инструкций запрещает агенту использовать самореферентные конструкции (мета-заглушки вида "Экран 1 рассматривает тему...").

### 3.3. Фаза В: Синтез тестовых метрик (Tester Agent)
Агент генерирует валидные контрольные вопросы (MCQA). Введены строгие процедурные ограничения на генерацию "рекурсивных" вопросов, требуя исключительно фактологической (Knowledge-Based) проверки усвоенного материала.

### 3.4. Асинхронная устойчивость (V4 Pipeline)
Во избежание состояния гонки (Race Condition), характерного для параллельного выполнения LLM-промптов в Node.js, внедрен синхронный конвейер *V4 Pipeline*. Парадигма Sequential Execution (строго последовательные await-блоки) гарантирует отсутствие таймаутов модели (OOM или Timeout Exceptions) и предотвращает утрату сгенерированных узлов.

## 4. Слой трансформации и упаковки (SCORM Builder)

Утвержденная JSON-модель данных трансформируется в валидный пакет по спецификации SCORM 1.2:
1. Рендеринг статических HTML-узлов на базе внедренных шаблонов.
2. Имплементация SCORM API (LMSInitialize, LMSCommit, LMSFinish) через инъекцию JavaScript-адаптера.
3. Генерация манифеста `imsmanifest.xml` с иерархическим описанием ресурсов.
4. Сжатие артефактов в буфер ZIP.

---

## 5. Инновация гибридной интеграции с LMS (Headless API Client)

Основным научным и инженерным новшеством системы является обход функциональной ограниченности спецификации SCORM в части сохранения телеметрии тестирования.

### 5.1. Дихотомия выгрузки
Система физически разделяет образовательную компоненту и контрольно-измерительную:
* Обучающие SCO-экраны интегрируются посредством стандартного импорта SCORM-пакета в LMS.
* Финальный тест **создается нативно** в модуле "Упражнения" (Exercises) LMS Chamilo.

### 5.2. Эмуляция контроллера (Headless Interaction)
Модуль `Chamilo Client` выступает в роли программного агента-автоматизатора (Headless Admin). Он авторизуется в LMS посредством управления Session Cookies и транслирует JSON-структуру финального теста в HTTP POST-запросы `exercise_admin.php`.

### 5.3. Динамический обход защитных механизмов (CSRF Token Override)
При программном присоединении созданного нативного теста к концу SCORM-траектории (Learning Path) API Chamilo использует временной токен `post_time` (эквивалент базового механизма защиты CSRF на стороне сервера на базе `time()`). 
Для успешной транзакции наш модуль осуществляет парсинг HTTP-заголовка `Date` сервера назначения, выполняет детерминированную калибровку локального времени скрипта в соответствии с *серверным временем (Server Epoch Time)* и осуществляет бесшовную пост-привязку (linking).

---

## 6. Заключение
Архитектура **SCORM-Chamilo Generator** демонстрирует высокоэффективный подход к применению LLM в закрытых корпоративных контурах. 
Интеграция RAG с алгоритмической предварительной очисткой обеспечивает 100% отслеживаемость фактов. Мультиагентный синхронный конвейер ликвидирует аппаратные состояния гонки. Наконец, уникальная стратегия гибридной выгрузки (SCORM Artifact + Native LMS Test) посредством Headless API обеспечивает пользователя полным спектром инструментов педагогической аналитики.
